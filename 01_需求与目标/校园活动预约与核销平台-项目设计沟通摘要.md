# 校园活动预约与优惠券核销平台：项目设计沟通摘要

生成日期：2026-07-19

## 1. 项目定位

项目名称暂定为：校园活动预约与优惠券核销平台。

该项目定位为 AI 应用实习方向的后端支撑项目，不作为主打 AI 项目，而是用于证明具备扎实的 Java 后端工程能力。后续可以扩展为校园 AI 应用、RAG 活动推荐、智能问答等系统的后端基础，但第一轮 MVP 不强行加入 AI 功能。

项目参考黑马点评的技术难度，但业务不能照搬黑马点评。核心业务应围绕校园活动预约、预约码核销、优惠券领取和秒杀展开，突出个人特色。

## 2. 已确认需求决策

1. 投递定位：AI 应用实习的后端支撑项目。
2. 登录方式：先使用手机号 + 密码，不做短信验证码。
3. 权限方案：先使用数据库内置角色，用户表保留 role 字段，支持 USER / ADMIN。
4. 第一轮 MVP：先不做优惠券，第一轮结束后再决定是否继续做优惠券、秒杀和 RabbitMQ。
5. 项目目录：沿用之前 WorkSpace 规范，建议目录名为 project003校园活动预约与核销平台。

## 3. 需求假设

1. 该项目主要用于 Java 后端实习简历展示。
2. MVP 优先证明 Spring Boot 3、MySQL 建模、接口设计、JWT 鉴权、Redis 缓存和库存控制能力。
3. 管理端先只做后端接口，不做前端后台页面。
4. 第一版优先打通完整业务闭环，而不是堆砌大量 Redis 功能。
5. 后续代码开发必须遵循小步推进，每次只完成一个可验证阶段。

## 4. MVP 范围

第一轮 MVP 只做以下核心模块：

1. 用户模块：注册、登录、JWT 登录态、当前用户信息。
2. 活动模块：活动分类、活动列表、活动详情、活动状态。
3. 活动预约模块：用户预约活动、取消预约、一人一单、防止超卖。
4. 核销模块：预约成功后生成预约码，管理员使用预约码核销。
5. Redis 缓存模块：活动详情缓存、缓存空值防穿透、TTL 随机化。
6. 基础工程模块：统一响应、统一异常、参数校验、Knife4j 接口文档。

第一轮暂不做：

1. 优惠券领取。
2. 秒杀优惠券。
3. RabbitMQ 异步落库。
4. BitMap 签到。
5. GEO 附近活动。
6. 排行榜。
7. 复杂 RBAC 权限系统。

## 8. Redis Key 和缓存策略

### MVP 必做 Redis Key

1. cache:activity:{id}
   - 类型：String JSON
   - 作用：缓存活动详情。

2. cache:activity:null:{id}
   - 类型：String
   - 作用：缓存不存在的活动 ID，防止缓存穿透。

3. lock:activity:reserve:{activityId}
   - 类型：Redisson Lock
   - 作用：后续用于预约库存扣减锁。MVP 可先用 MySQL 条件扣减。

### 后续秒杀阶段 Redis Key

1. quota:activity:{activityId}
   - 类型：String
   - 作用：活动剩余名额。

2. reserve:user:{activityId}
   - 类型：Set
   - 作用：记录已预约用户，防止重复预约。

3. seckill:coupon:stock:{couponId}
   - 类型：String
   - 作用：秒杀优惠券库存。

4. seckill:coupon:users:{couponId}
   - 类型：Set
   - 作用：记录已抢券用户。

### 缓存策略

1. 活动详情使用 Cache Aside 模式：先查 Redis，未命中查 MySQL，再写 Redis。
2. 不存在的活动写入空值缓存，TTL 2 到 5 分钟。
3. 正常活动详情缓存 TTL 增加随机秒数，降低同一时间大量过期的风险。
4. 预约库存 MVP 先使用 MySQL 条件扣减：available_quota > 0。
5. 秒杀阶段再升级为 Redis + Lua 原子判断库存和重复预约。
6. 异步阶段通过 RabbitMQ 创建预约记录或优惠券领取记录。

## 9. 开发阶段计划和验收标准

### 阶段 0：项目初始化

目标：创建 Spring Boot 3 项目骨架，接入 MySQL、MyBatis-Plus、Knife4j。

验收标准：

1. 项目可以本地启动。
2. 测试接口可以返回成功。
3. Knife4j 接口文档可以访问。
4. MySQL 连接正常。

### 阶段 1：用户登录

目标：完成注册、登录、JWT 鉴权和当前用户信息接口。

验收标准：

1. 用户可以注册。
2. 登录成功后返回 token。
3. 不带 token 访问受保护接口失败。
4. 带 token 可以访问 /api/users/me。

### 阶段 2：活动基础功能

目标：完成活动分类、活动列表、活动详情和管理员活动管理接口。

验收标准：

1. 管理员可以创建活动。
2. 用户可以分页查询活动列表。
3. 用户可以查看活动详情。
4. 活动状态可以正确控制展示和预约。

### 阶段 3：活动预约 MVP

目标：完成用户预约活动、库存扣减和一人一单。

验收标准：

1. 用户预约成功后生成预约记录。
2. 同一用户重复预约同一活动失败。
3. 活动名额为 0 时预约失败。
4. 并发场景下不出现 available_quota 小于 0。

### 阶段 4：取消预约与核销

目标：完成用户取消预约和管理员核销预约码。

验收标准：

1. 用户可以取消未核销预约。
2. 取消后预约状态变为 CANCELED。
3. 管理员可以核销有效预约码。
4. 已核销预约不能重复核销。
5. 错误预约码核销失败。

### 阶段 5：Redis 缓存

目标：活动详情接入 Redis 缓存。

验收标准：

1. 第一次查询活动详情时查询 MySQL 并写入 Redis。
2. 第二次查询同一活动详情时命中 Redis。
3. 查询不存在活动时写入空值缓存。
4. Redis Key 有合理 TTL。

### 阶段 6：第一轮复盘和二次决策

目标：第一轮 MVP 完成后暂停，重新决定是否进入优惠券、秒杀、RabbitMQ 阶段。

验收标准：

1. 第一轮主链路可以完整演示。
2. 接口文档完整。
3. 数据库数据状态正确。
4. 明确下一阶段是否加入优惠券和秒杀。

## 11. 后续协作规则

1. 每次只推进一个小阶段。
2. 编码前必须先说明修改目标和验收方式。
3. 修改代码前必须先阅读相关文件。
4. 只修改当前阶段需要的内容，不做无关重构。
5. 不新增生产依赖，除非先确认。
6. 每个阶段都必须运行验证命令或说明无法验证的原因。
