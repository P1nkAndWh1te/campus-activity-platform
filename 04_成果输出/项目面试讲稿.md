# 校园活动预约与核销平台 - 项目面试讲稿

## 1 分钟项目介绍

我做的是一个校园活动预约与核销平台，主要面向校园讲座、比赛、社团活动等场景。项目后端使用 Spring Boot 3 和 MyBatis-Plus，数据库使用 MySQL，缓存使用 Redis，消息队列使用 RabbitMQ。

项目核心流程是：管理员创建活动并设置名额，普通用户登录后可以查询活动并预约，预约成功后系统生成预约码；用户到现场后，管理员可以根据预约码查询预约并完成核销。

我在这个项目里重点处理了三个问题。第一是重复预约，通过预约表上的用户 ID 和活动 ID 唯一索引兜底限制一人一单。第二是名额扣减，预约时使用 `available_quota > 0` 的条件更新，避免库存被扣成负数。第三是核心链路可验证，我写了自动化接口验收脚本，覆盖注册、登录、预约、重复预约拦截、核销、重复核销拦截和取消预约等流程。

另外，项目还使用 JWT 做登录鉴权，使用 Redis 缓存活动详情，并通过 RabbitMQ 在预约成功后异步写入通知记录。

## 30 秒精简版

这个项目是一个基于 Spring Boot 3 的校园活动预约与核销后端系统，支持用户登录、活动查询、活动预约、取消预约、管理员核销等功能。核心亮点是通过唯一索引限制重复预约，通过条件更新避免名额超卖，通过事务保证预约记录和名额扣减一致。项目还使用 JWT 做鉴权，Redis 缓存活动详情，RabbitMQ 异步处理预约通知，并提供自动化接口验收脚本验证核心链路。

## 项目主流程

```text
用户登录 -> 获取 JWT
用户查询活动列表 -> 查询活动详情
用户预约活动 -> 条件扣减名额 -> 写入预约记录 -> 生成预约码
事务提交后 -> 发送 RabbitMQ 消息 -> 异步写入通知记录
管理员输入预约码 -> 查询预约 -> 核销 -> 写入核销记录
```

## 高频追问 1：你是怎么防止用户重复预约的？

我用了两层处理。

第一层是业务层判断，预约时会根据当前用户 ID 和活动 ID 查询是否已经存在预约记录。

第二层是数据库兜底，在 `activity_reservation` 表上建立了 `(user_id, activity_id)` 唯一索引。即使并发请求同时进入，数据库也能保证同一个用户对同一个活动只能插入一条预约记录。

如果插入时触发唯一索引异常，业务层会返回“已预约过该活动”的错误提示。

## 高频追问 2：你是怎么防止名额超卖的？

预约时不是先查询剩余名额再在 Java 代码里减一，而是直接在数据库执行条件更新：

```sql
UPDATE activity
SET available_quota = available_quota - 1
WHERE id = ? AND available_quota > 0;
```

如果影响行数是 1，说明扣减成功；如果影响行数是 0，说明名额不足，预约失败。这样可以避免多个请求同时读到相同库存后都成功扣减的问题。

## 高频追问 3：为什么还要加事务？

预约流程里至少有两个关键写操作：扣减活动名额、插入预约记录。

如果扣减名额成功后，插入预约记录失败，就会造成名额少了但用户没有预约成功。所以我在预约方法上加了 `@Transactional`，让这两个操作处于同一个事务中。只要后续插入预约记录失败，前面的库存扣减也会回滚。

## 高频追问 4：JWT 登录鉴权是怎么做的？

用户登录成功后，服务端会生成 JWT，里面包含用户 ID 和角色。前端后续请求时在请求头携带：

```text
Authorization: Bearer token
```

后端拦截器会检查请求头是否存在 token，并验证 token 是否有效。验证通过后，把用户 ID 和角色写入 `UserContext`，业务代码就可以通过 `UserContext` 获取当前登录用户，判断是否有管理员权限。

## 高频追问 5：为什么使用 Redis？

活动详情是典型的读多写少数据。用户在预约前会频繁查看活动详情，如果每次都查 MySQL，数据库压力会比较大。

所以我使用 Redis 做活动详情缓存，采用 Cache-Aside 模式：先查 Redis，未命中再查 MySQL，然后把结果写回 Redis。对于不存在的活动，也会写入一个短 TTL 的空值缓存，减少恶意请求不存在 ID 导致的缓存穿透问题。

同时正常缓存设置了基础 TTL 加随机抖动，降低大量 key 同时过期的风险。

## 高频追问 6：为什么使用 RabbitMQ？

预约成功后，系统需要写入通知记录。这个动作不应该阻塞预约主流程，因为用户最关心的是预约是否成功。

所以我在预约事务提交后发送 RabbitMQ 消息，消费者异步写入通知记录。这样预约接口只负责核心业务，通知写入失败也不会直接影响用户预约成功结果。

这里我刻意放在事务提交后发送消息，避免事务回滚了但消息已经发出去，导致通知和真实预约状态不一致。

## 高频追问 7：核销流程怎么保证不会重复核销？

管理员核销时，先根据预约码查询预约记录，然后检查预约状态：

- `CANCELED`：说明预约已取消，不能核销。
- `VERIFIED`：说明已经核销过，不能重复核销。
- `RESERVED`：说明预约有效，可以核销。

核销成功后，会把预约状态改为 `VERIFIED`，并写入一条核销记录。

## 高频追问 8：你这个项目和普通 CRUD 有什么区别？

这个项目不是只做增删改查，核心在于预约业务链路上的一致性处理。

比如预约活动时，要同时考虑一人一单、库存不能为负、预约记录和库存扣减要一致、预约成功后还有异步通知。核销时，也要考虑预约码是否有效、是否已取消、是否已经核销。

所以项目重点不是接口数量，而是围绕真实业务流程处理边界情况，并且用脚本把这些核心链路验证出来。

## 高频追问 9：如果让你继续优化，你会怎么做？

我会按优先级做四件事：

1. 补充单元测试和集成测试，尤其是预约、取消、核销这些核心服务。
2. 使用 Apifox 或 Postman 保存接口测试集合，方便演示和回归。
3. 增加简单前端页面或管理端页面，让项目更容易展示。
4. 如果需要模拟高并发预约，再考虑引入 Redis 预扣库存、Lua 脚本或消息队列异步落库。

我不会一开始就堆复杂架构，因为当前 MVP 的目标是先把主业务链路跑通并验证清楚。

## 高频追问 10：项目目前有什么不足？

目前第一版主要是后端 MVP，没有前端页面，也没有做线上部署。

权限控制目前是简单的 `role` 字段判断，没有实现完整 RBAC。

RabbitMQ 目前只用于预约成功通知，没有扩展到秒杀或异步落库。

另外，虽然有自动化接口验收脚本，但还可以继续补充 Apifox 测试集合、接口截图和更完整的自动化测试。

## 面试时不要主动夸大的点

不要说：

- 我做了高并发秒杀系统。
- 我用 Lua 保证了 Redis 原子扣库存。
- 我做了完整分布式事务。
- 我完成了前后端完整项目。
- 项目已经线上部署。

可以说：

- 当前项目是后端 MVP，重点验证预约与核销主链路。
- 当前通过 MySQL 唯一索引和条件更新处理重复预约和名额扣减问题。
- 如果继续扩展到高并发场景，可以考虑 Redis 预扣库存、Lua 脚本、限流和异步削峰。

## 面试官追问时的底线表达

如果被问到没有实现的内容，可以这样回答：

这个版本我没有直接上复杂高并发方案，因为项目当前目标是先完成一个可运行、可验证的后端 MVP。我优先把数据库约束、事务、条件扣减、JWT 鉴权、Redis 缓存和 RabbitMQ 通知这些基础链路做扎实。后续如果扩展到抢票或秒杀场景，我会再引入 Redis 原子扣减、限流、消息队列削峰和更完整的压测。

## 建议背熟的 5 句话

1. 这个项目的核心不是接口数量，而是预约和核销链路的一致性。
2. 重复预约靠业务判断加数据库唯一索引兜底。
3. 名额扣减使用 `available_quota > 0` 条件更新，避免库存被扣成负数。
4. JWT 拦截器负责解析用户身份和角色，业务层通过 `UserContext` 获取当前用户。
5. Redis 用于活动详情缓存，RabbitMQ 用于预约成功后的通知异步化。
